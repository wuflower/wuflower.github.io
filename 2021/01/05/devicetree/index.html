Android项目Device Tree深入理解（一）

前言

一条清晰的概念应该包含3个要素what（是什么）、why（为什么）、how（怎么做） ，只有明确了这3要素才能完全掌握这个概念。设备树DT是做底层驱动的人经常会接触到的，我工作到现在做的项目也一直在通过修改.dts或者.dtsi文件来bring up设备，但对于我来说之前都是按照老项目的模板或者网上的一些文档、平台release的一些配置方法去配置，按照之前的定义对于Device Tree我还只知道如何配置（也就是how），需要知其然知其所以然。

概述

在Linux3.x之前的内核源码中，存在大量对板级细节信息描述的hard code。这些代码充斥在/arch/arm/plat-xxx和/arch/arm/mach-xxx目录。为了解决这个问题而引入设备树。Device Tree 是一种描述硬件的数据结构，可以把嵌入式系统资源抽象成一颗树形结构，可以直观查看系统资源分布；内核可以识别这棵树，并根据它展开出Linux内核的platfrom_device等。它通过bootloader将硬件资源传给内核，使得内核和硬件资源描述相对独立。

设备树的主要优势：对于同一SOC的不同主板，只需更换设备树文件.dtb即可实现不同主板的无差异支持，而无需更换内核文件。

Android项目中的Device Tree

基本结构

对于目前主流的arm x86_64架构，Android项目中DT的路径在kernel/arch/arm64/boot/dtc/(SoC厂商)qcom或者mediatek等/。在Android中DT被分割成两个部分：

  • 主DT：由 SoC 供应商提供的仅限 SoC 访问的部分和默认配置。定义了SOC级配置，如设备时钟、中断等资源；

  • 叠加DT：由原始设计制造商 (ODM)/原始设备制造商 (OEM) 提供的设备专用配置。定义了板级配置，包含一些板级差异信息。

用我们项目的实际情况通俗的解释就是：Soc级配置就是指高通或者MTK这样的平台默认的一些配置，board级配置就是我们不同项目的配置。

Device Tree的源文件叫Device Tree Sourcefile（DTS），在项目中以.dts和.dtsi格式的文件存在。由于一个SoC可能对应多个机器（一个平台对应多款产品），势必这些.dts文件需包含许多共同的部分，Linux内核为了简化，把SoC公用的部分或者多个机器共同的部分一般提炼为.dtsi，类似于C语言的头文件，一个设备树由1个.dts文件+n个.dtsi文件组成。

以高通6115平台为例，一个项目(xxx)的设备树由以下两部分组成

  • 主DT：包含begal.dts和它include的其它所有.dtsi文件

  • 叠加DT：包含xxx.dts和它include的其它所有.dtsi文件

主DT和叠加DT

dtb和dtbo

dtb：device tree blob 设备树二进制文件

dtbo：device tree blob overlay

构建主DT需要执行以下操作：

  1. 将主DT.dts 编译为 .dtb 文件。

  2. 将 .dtb 文件刷写到引导加载程序在运行时可访问的分区。

如需构建叠加 DT需要执行以下操作：

  1. 将叠加DT.dts 编译为 .dtbo 文件。虽然此文件格式与格式设为扁平化设备树的 .dtb 文件相同，但是用不同的文件扩展名可以将其与主 DT 区分开来。

  2. 将 .dtbo 文件刷写到引导加载程序在运行时可访问的分区。

也就是说主DT和叠加DT编译后产生的文件并不相同，对应关系如下

主DT → 编译 → .dtb 文件

叠加DT → 编译 → .dtbo 文件

dtbo 最后的o是overlay的意思，dtb在内核启动的时候会被加载，有些功能经常改变，或者希望动态加载，那么可以使用dtbo。这个文件不需要像dtb那样在引导的时候就需要准备好，放在预定好的位置。dtbo是可以后期动态的修改和加载，dtbo可以用在灵活多变的模块中。
